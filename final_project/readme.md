# Организация защищённой корпоративной сети с филиалами и ЦОД

## Аннотация

В рамках проектной работы была спроектирована и реализована лабораторная модель защищённой корпоративной сети организации с центральным офисом, двумя удалёнными филиалами и корпоративным центром обработки данных.

Лабораторная среда построена в EVE-NG. В проекте используются технологии VLAN, HSRP, OSPF, EIGRP, BGP, DMVPN over IPSec, NAT/PAT, ACL, DHCP, NTP и SSH.  
Основная цель проекта — показать, как можно построить распределённую корпоративную сеть с резервированием, динамической маршрутизацией и защищённой передачей данных между площадками.

В финальной версии лаборатории для упрощения реализации на каждой офисной площадке используется по два access-коммутатора. Такая схема сохраняет демонстрацию основных технологий: VLAN, trunk, HSRP, OSPF, BGP, DMVPN, EIGRP и ACL, но делает лабораторию проще для сборки и защиты.

## 2. Цель проекта

Цель проекта — разработать и реализовать защищённую распределённую корпоративную сеть, включающую центральный офис, удалённые филиалы и корпоративный ЦОД, с использованием современных технологий маршрутизации, резервирования и защиты трафика.

---

## 3. Задачи проекта

В ходе выполнения проекта решались следующие задачи:

1. Спроектировать логическую топологию корпоративной сети.
2. Разработать адресный план для центрального офиса, филиалов, ЦОД, провайдеров и магистральной сети.
3. Разделить локальные сети офисов на VLAN по функциональному назначению.
4. Настроить маршрутизацию между VLAN.
5. Реализовать резервирование шлюзов пользователей с помощью HSRP.
6. Настроить OSPF внутри офисных площадок.
7. Настроить BGP между офисами, провайдерами, магистральной сетью и ЦОД.
8. Настроить iBGP внутри магистральной сети провайдера.
9. Организовать защищённые DMVPN over IPSec-туннели между площадками.
10. Настроить EIGRP поверх DMVPN для обмена корпоративными маршрутами.
11. Настроить NAT/PAT для выхода внутренних сетей во внешнюю транспортную среду.
12. Исключить корпоративный VPN-трафик из NAT.
13. Настроить ACL для ограничения доступа между пользовательскими сегментами и серверами ЦОД.
14. Настроить базовые инфраструктурные сервисы DHCP, NTP и SSH.
15. Выполнить проверку работоспособности сети.
16. Подготовить итоговую документацию проекта.

## 4. Общая характеристика проектируемой сети

Проектируемая сеть состоит из нескольких независимых логических областей:

| Область | Назначение |
|---|---|
| Центральный офис Москва | Основная офисная площадка компании, DMVPN Hub |
| Филиал Казань | Удалённая офисная площадка |
| Филиал Новосибирск | Удалённая офисная площадка |
| Корпоративный ЦОД | Размещение серверов и инфраструктурных сервисов |
| ISP | Провайдеры для подключения офисов к внешней транспортной сети |
| Backbone Provider | Магистральная сеть провайдера, связывающая ISP и ЦОД |

Локальные сети офисов построены по иерархической модели:

```text
EDGE-уровень       — подключение к провайдерам, NAT, BGP, DMVPN
DIST-уровень       — маршрутизация VLAN, HSRP, OSPF
ACCESS-уровень     — подключение конечных устройств
```

В финальной лаборатории на каждой офисной площадке используется по два access-коммутатора:

```text
HQ:  HQ-ACC1,  HQ-ACC2
KZN: KZN-ACC1, KZN-ACC2
NSK: NSK-ACC1, NSK-ACC2
```

Такой вариант был выбран для упрощения сборки в EVE-NG и уменьшения количества физических линков, при этом вся основная логика проекта сохраняется.

- DMVPN over IPSec-сеть между площадками.

```text
                 +-------------------+
                 | Корпоративный ЦОД |
                 |     AS 65550      |
                 +---------+---------+
                           |
                           |
                 +---------+---------+
                 | Backbone Provider |
                 |     AS 65000      |
                 +----+----+----+----+
                      /      |      \
                     /       |       \
              ISP-HQ      ISP-KZN    ISP-NSK
                |            |          |
         +------+---+  +-----+----+ +---+------+
         |  Москва  |  |  Казань | | Новосибирск |
         | AS 65100 |  |AS 65200 | |  AS 65300   |
         +----------+  +---------+ +-------------+
```

Корпоративный трафик между площадками передаётся через DMVPN over IPSec.  
BGP используется для построения транспортной связности между автономными системами.

## 7. Автономные системы

### 7.1. Что такое AS

AS, или Autonomous System, — это автономная система: группа сетей, находящаяся под единым административным управлением и использующая единую политику маршрутизации.

AS-номера используются протоколом BGP. Благодаря AS-номерам BGP понимает, какие маршруты принадлежат разным административным доменам, и строит междоменную маршрутизацию.

В проекте AS-номера используются для имитации взаимодействия:

- корпоративных площадок;
- провайдеров;
- магистральной сети;
- корпоративного ЦОД.

## 13. Настройка VLAN и access-уровня

На access-коммутаторах созданы VLAN:

```cisco
vlan 10
 name USERS
vlan 20
 name DEV
vlan 30
 name MANAGEMENT
vlan 40
 name FINANCE
```

Uplink-порт `Ethernet0/0` настроен как trunk.  
Пользовательские порты настроены как access.

Пример для HQ-ACC1:

```cisco
interface Ethernet0/0
 description TRUNK-TO-HQ-DIST1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30,40

interface Ethernet0/1
 description TO-HQ-USERS-1
 switchport mode access
 switchport access vlan 10

interface Ethernet0/2
 description TO-HQ-USERS-2
 switchport mode access
 switchport access vlan 10
```

Для HQ-ACC2 пользовательские порты отнесены к VLAN 20:

```cisco
interface Ethernet0/1
 description TO-HQ-DEV-1
 switchport mode access
 switchport access vlan 20

interface Ethernet0/2
 description TO-HQ-DEV-2
 switchport mode access
 switchport access vlan 20
```

Аналогичная логика применена для Казани и Новосибирска.

## 14. Настройка HSRP

HSRP используется для резервирования шлюзов VLAN между двумя DIST-устройствами на каждой площадке.

### 14.1. Принцип работы

На каждом VLAN создаётся виртуальный IP-адрес, который используется конечными устройствами как шлюз по умолчанию.

Например, в Москве для VLAN 10:

```text
HQ-DIST1: 10.100.10.1
HQ-DIST2: 10.100.10.2
HSRP VIP: 10.100.10.254
```

Пользовательские ПК используют шлюз `10.100.10.254`.  
Если активный DIST выходит из строя, второй DIST принимает роль Active, а шлюз остаётся тем же.

### 14.2. Распределение ролей HSRP

#### Москва

|---:|---|---|---|
| 10 | HQ-DIST1 | HQ-DIST2 | 10.100.10.254 |
| 20 | HQ-DIST2 | HQ-DIST1 | 10.100.20.254 |
| 30 | HQ-DIST1 | HQ-DIST2 | 10.100.30.254 |
| 40 | HQ-DIST2 | HQ-DIST1 | 10.100.40.254 |

#### Казань

| VLAN | Active | Standby | VIP |
|---:|---|---|---|
| 10 | KZN-DIST1 | KZN-DIST2 | 10.110.10.254 |
| 20 | KZN-DIST2 | KZN-DIST1 | 10.110.20.254 |
| 30 | KZN-DIST1 | KZN-DIST2 | 10.110.30.254 |
| 40 | KZN-DIST2 | KZN-DIST1 | 10.110.40.254 |

#### Новосибирск

| VLAN | Active | Standby | VIP |
|---:|---|---|---|
| 10 | NSK-DIST1 | NSK-DIST2 | 10.120.10.254 |
| 20 | NSK-DIST2 | NSK-DIST1 | 10.120.20.254 |
| 30 | NSK-DIST1 | NSK-DIST2 | 10.120.30.254 |
| 40 | NSK-DIST2 | NSK-DIST1 | 10.120.40.254 |

## 15. Настройка OSPF

OSPF используется внутри офисов между EDGE- и DIST-устройствами.

### 15.1. Назначение OSPF

OSPF решает следующие задачи:

- обеспечивает обмен маршрутами внутри площадки;
- передаёт маршруты до VLAN-сетей;
- обеспечивает связность между EDGE и DIST;
- позволяет EDGE-устройствам узнавать локальные сети площадки;
- создаёт базу для redistribution в EIGRP.

### 15.2. Область OSPF

Во всех офисах используется area 0:

```cisco
router ospf 1
 router-id <Loopback1>
 passive-interface default
 no passive-interface Ethernet0/0
 no passive-interface Ethernet0/2
 no passive-interface Ethernet0/3
```

Пользовательские VLAN являются passive-interface, чтобы маршрутизатор не пытался устанавливать OSPF-соседство с конечными устройствами.

### 15.3. Проверка OSPF

Команда:

```cisco
show ip ospf neighbor
```

Ожидаемый результат:

```text
Соседства между EDGE и DIST находятся в состоянии FULL.
```

## 16. Настройка BGP

BGP используется для маршрутизации между автономными системами.

### 16.1. Где используется BGP

| Участок | Тип BGP |
|---|---|
| EDGE офисов — ISP | eBGP |
| ISP — Backbone | eBGP |
| Backbone внутри AS 65000 | iBGP |
| DC-EDGE1 — BB-R1 | eBGP |
| EDGE1 — EDGE2 внутри площадки | iBGP |

### 16.2. Назначение BGP в проекте

BGP в проекте используется не для передачи пользовательского корпоративного трафика напрямую, а для построения внешней транспортной связности.

Эта транспортная связность нужна для того, чтобы внешние адреса EDGE-устройств были достижимы между площадками. Поверх этой связности строятся DMVPN over IPSec-туннели.

### 16.3. BGP на Backbone

В Backbone Provider маршрутизаторы BB-R1–BB-R4 находятся в AS 65000.

Внутри AS 65000 используется iBGP. Для iBGP-соседств используются loopback-адреса:

```text
BB-R1 100.64.15.1
BB-R2 100.64.15.2
BB-R3 100.64.15.3
BB-R4 100.64.15.4
```

Для доступности loopback-адресов внутри Backbone используется OSPF.

### 16.4. Проверка BGP

Команда:

```cisco
show ip bgp summary
```

Успешное BGP-соседство отображается числом в колонке `State/PfxRcd`.  
Если в колонке отображается `Idle`, `Active` или `Connect`, соседство не установлено.

Пример успешного состояния:

```text
Neighbor        AS      State/PfxRcd
100.64.15.2     65000   0
100.64.101.2    64501   0
```

## 17. Настройка DMVPN over IPSec

### 17.1. Назначение DMVPN

DMVPN используется для создания защищённой корпоративной WAN-сети поверх внешней транспортной инфраструктуры.

Преимущество DMVPN в том, что не требуется вручную создавать отдельный GRE/IPSec-туннель между каждой парой площадок. Spoke-устройства регистрируются на Hub-устройствах, а затем могут устанавливать динамические соединения.

### 17.2. Роли устройств

| Устройство | Роль |
|---|---|
| HQ-EDGE1 | Hub 1 |
| HQ-EDGE2 | Hub 2 |
| KZN-EDGE1 | Spoke |
| KZN-EDGE2 | Spoke |
| NSK-EDGE1 | Spoke |
| NSK-EDGE2 | Spoke |
| DC-EDGE1 | Spoke |

### 17.3. Используемые компоненты

| Компонент | Назначение |
|---|---|
| GRE multipoint | Туннелирование маршрутизируемого трафика |
| NHRP | Регистрация spoke-узлов на hub |
| IPSec | Шифрование трафика |
| ISAKMP/IKE | Согласование параметров шифрования |
| Tunnel0 | Логический интерфейс DMVPN |

### 17.4. Параметры IPSec

В проекте используются следующие параметры:

```cisco
crypto isakmp policy 10
 encr aes 256
 hash sha256
 authentication pre-share
 group 14
 lifetime 86400
```

Transform-set:

```cisco
crypto ipsec transform-set DMVPN-TS esp-aes 256 esp-sha256-hmac
 mode transport
```

IPSec profile:

```cisco
crypto ipsec profile DMVPN-PROFILE
 set transform-set DMVPN-TS
```

### 17.5. Проверка DMVPN

Команды:

```cisco
show dmvpn
show crypto isakmp sa
show crypto ipsec sa
```

Ожидаемый результат:

- spoke-устройства отображаются на hub;
- ISAKMP-сессии активны;
- в IPSec SA увеличиваются счётчики encrypted/decrypted packets.

## 18. Настройка EIGRP поверх DMVPN

### 18.1. Назначение EIGRP

EIGRP используется поверх DMVPN для обмена корпоративными маршрутами между площадками.

Именно EIGRP передаёт маршруты до сетей:

```text
10.100.0.0/16
10.110.0.0/16
10.120.0.0/16
10.150.0.0/16
```

### 18.2. EIGRP AS

В проекте используется:

```cisco
router eigrp 100
```

### 18.3. Анонсируемые маршруты

| Площадка | Анонсируемый маршрут |
|---|---|
| Москва | 10.100.0.0/16 |
| Казань | 10.110.0.0/16 |
| Новосибирск | 10.120.0.0/16 |
| ЦОД | 10.150.0.0/16 |

### 18.4. Redistribution

На EDGE-устройствах выполняется redistribution между OSPF и EIGRP:

```cisco
router ospf 1
 redistribute eigrp 100 subnets

router eigrp 100
 redistribute ospf 1 metric 10000 100 255 1 1500
```

Это нужно для того, чтобы:

- локальные VLAN-сети попадали в DMVPN;
- удалённые маршруты из DMVPN попадали во внутреннюю сеть офиса.

### 18.5. Проверка EIGRP

Команды:

```cisco
show ip eigrp neighbors
show ip route eigrp
```

Ожидаемый результат:

- EDGE-устройства видят EIGRP-соседей через Tunnel0;
- в таблице маршрутизации появляются маршруты до удалённых площадок.

## 19. Настройка NAT/PAT

### 19.1. Назначение NAT

NAT/PAT используется для выхода внутренних сетей во внешнюю транспортную сеть через внешний интерфейс EDGE-устройства.

Пример:

```text
10.100.10.101 → NAT на адрес внешнего интерфейса HQ-EDGE1
```

### 19.2. Исключение VPN-трафика из NAT

Корпоративный трафик между площадками не должен NAT'иться.  
Если NAT применится к трафику между внутренними сетями, DMVPN-маршрутизация может работать некорректно.

Поэтому в ACL для NAT используются deny-правила для корпоративных сетей и permit для остального трафика.

Пример для Москвы:

```cisco
ip access-list extended NAT-HQ
 deny ip 10.100.0.0 0.0.255.255 10.110.0.0 0.0.255.255
 deny ip 10.100.0.0 0.0.255.255 10.120.0.0 0.0.255.255
 deny ip 10.100.0.0 0.0.255.255 10.150.0.0 0.0.255.255
 permit ip 10.100.0.0 0.0.255.255 any
```

### 19.3. Проверка NAT

Команды:

```cisco
show ip nat translations
show ip nat statistics
show access-lists NAT-HQ
```

Ожидаемый результат:

- внешний трафик NAT'ится;
- трафик в корпоративные сети через DMVPN не NAT'ится.

## 20. Настройка ACL

### 20.1. Назначение ACL

ACL используются для ограничения доступа между пользовательскими сегментами и серверными ресурсами.

В проекте применена следующая логика:

| Сегмент | Доступ |
|---|---|

| MANAGEMENT | Используется для администрирования оборудования |

### 20.2. Серверы ЦОД

| Сервер | IP |
|---|---|
| APP-SRV | 10.150.100.10 |
| DB-SRV | 10.150.100.20 |
| NTP-DHCP | 10.150.100.30 |

### 20.3. Пример ACL для USERS

```cisco
ip access-list extended HQ-USERS-IN
 permit ip 10.100.10.0 0.0.0.255 host 10.150.100.10
 deny ip 10.100.10.0 0.0.0.255 host 10.150.100.20
 permit ip 10.100.10.0 0.0.0.255 host 10.150.100.30
 permit ip 10.100.10.0 0.0.0.255 any
```

ACL применяется на вход subinterface VLAN:

```cisco
interface Ethernet0/1.10

```

## 21. DHCP

DHCP используется для автоматической выдачи IP-адресов конечным устройствам.

В лаборатории DHCP-пулы настроены на DIST-устройствах.

Пример DHCP для VLAN USERS в Москве:

```cisco
ip dhcp pool HQ-USERS
 network 10.100.10.0 255.255.255.0
 default-router 10.100.10.254
 dns-server 10.150.100.30
```

Также для простоты проверки конечные устройства могут быть настроены статически.

Пример для VPCS:

```text
ip 10.100.10.101/24 10.100.10.254
save
```

## 27. Вывод

В результате выполнения проекта была построена защищённая распределённая корпоративная сеть с центральным офисом, двумя филиалами и корпоративным ЦОД.

В сети реализованы:

- логическое разделение пользователей с помощью VLAN;
- резервирование шлюзов VLAN с помощью HSRP;
- динамическая маршрутизация внутри площадок с помощью OSPF;
- внешняя маршрутизация между автономными системами с помощью BGP;
- защищённая связь между площадками через DMVPN over IPSec;
- обмен корпоративными маршрутами поверх DMVPN с помощью EIGRP;
- NAT/PAT для внешнего трафика;
- исключение VPN-трафика из NAT;
- базовые ACL для разграничения доступа;
- SSH для безопасного управления;
- DHCP и NTP как инфраструктурные сервисы.

Проведённые проверки подтвердили работоспособность основных компонентов проекта: VLAN, trunk, HSRP, OSPF, BGP, DMVPN, IPSec, EIGRP, NAT, ACL, SSH, NTP и механизмов отказоустойчивости.
